00022 /*
00023  * Moving GC Stack Rooting
00024  *
00025  * A moving GC may change the physical location of GC allocated things, even
00026  * when they are rooted, updating all pointers to the thing to refer to its new
00027  * location. The GC must therefore know about all live pointers to a thing,
00028  * not just one of them, in order to behave correctly.
00029  *
00030  * The classes below are used to root stack locations whose value may be held
00031  * live across a call that can trigger GC (i.e. a call which might allocate any
00032  * GC things). For a code fragment such as:
00033  *
00034  * Foo();
00035  * ... = obj->lastProperty();
00036  *
00037  * If Foo() can trigger a GC, the stack location of obj must be rooted to
00038  * ensure that the GC does not move the JSObject referred to by obj without
00039  * updating obj's location itself. This rooting must happen regardless of
00040  * whether there are other roots which ensure that the object itself will not
00041  * be collected.
00042  *
00043  * If Foo() cannot trigger a GC, and the same holds for all other calls made
00044  * between obj's definitions and its last uses, then no rooting is required.
00045  *
00046  * Several classes are available for rooting stack locations. All are templated
00047  * on the type T of the value being rooted, for which RootMethods<T> must
00048  * have an instantiation.
00049  *
00050  * - Rooted<T> declares a variable of type T, whose value is always rooted.
00051  *   Rooted<T> may be automatically coerced to a Handle<T>, below. Rooted<T>
00052  *   should be used whenever a local variable's value may be held live across a
00053  *   call which can allocate GC things or otherwise trigger a GC.
00054  *
00055  * - Handle<T> is a const reference to a Rooted<T>. Functions which take GC
00056  *   things or values as arguments and need to root those arguments should
00057  *   generally use handles for those arguments and avoid any explicit rooting.
00058  *   This has two benefits. First, when several such functions call each other
00059  *   then redundant rooting of multiple copies of the GC thing can be avoided.
00060  *   Second, if the caller does not pass a rooted value a compile error will be
00061  *   generated, which is quicker and easier to fix than when relying on a
00062  *   separate rooting analysis.
00063  */
00064

00262 /*
00263  * The *alleged* killer whale hack (see JS::Anchor<T> in jsapi.h) duplicated
00264  * here because using JS::Anchor causes clang to emit bad instructions.
00265  *
00266  * In exact-gc builds, Rooted<T> always keeps the T reachable. In non-exact-gc
00267  * builds, it does not, but conservative scanning usually picks up the slack.
00268  * However in the case where the Rooted pointer is no longer used, but some
00269  * subobject or malloc'd memory with the same lifetime may be used,
00270  * conservative scanning can fail. JSStableString's chars() method makes it
00271  * particularly attractive to use that way, so we use some voodoo to convince
00272  * the compiler to keep the string pointer on the stack for the full lifetime
00273  * of the Rooted<JSStableString *>.
00274  */

00276 struct RootSink<JSStableString *> {
00277     static void dispose(JSStableString *ptr) {
00278         JSStableString * volatile sink;
00279         sink = ptr;
00280     }
